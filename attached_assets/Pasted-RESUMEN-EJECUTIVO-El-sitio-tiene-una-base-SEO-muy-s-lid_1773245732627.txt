1. ERROR CRÍTICO: INCOHERENCIA DE DOMINIO
Hallazgo
El sitio usa dos dominios diferentes en distintas partes del código:

avisa.es → Usado en el motor SEO interno, sitemaps, canónicas dinámicas, y slugs
grupoavisa.com → Usado en index.html, Open Graph, Schema.org estático, llms.txt, og-image
Impacto (CRÍTICO)
Google ve dos sitios diferentes. Las canónicas del index.html apuntan a grupoavisa.com pero el motor SEO genera canónicas con avisa.es. Esto causa canibalización o ignorancia de canónicas.
Open Graph apunta a grupoavisa.com → Las imágenes sociales (og-image.jpg) no se resolverán si el dominio real es avisa.es.
Schema.org LocalBusiness y Organization referencian grupoavisa.com, pero los sitemaps apuntan a avisa.es. Los motores de búsqueda no pueden consolidar la entidad.
El archivo llms.txt tiene todas las URLs con grupoavisa.com pero el sitemap usa avisa.es.
Archivos afectados
Archivo	Dominio usado
index.html (líneas 23, 49, 52, 61, 68-190)	grupoavisa.com
public/llms.txt (130+ URLs)	grupoavisa.com
public/robots.txt (línea 68)	avisa.es
public/sitemap.xml (5 sub-sitemaps)	avisa.es
src/lib/seo-engine.ts (BASE_URL)	avisa.es
src/lib/seo/sitemap-engine.ts (BASE_URL)	avisa.es
src/hooks/useSEO.ts (canónicas)	avisa.es
src/hooks/useSEOAuto.ts (BASE_URL, og_image)	mezclado
src/lib/seo-service.ts (og_image fallback)	grupoavisa.com
Acción requerida
Decidir cuál es el dominio definitivo (avisa.es o grupoavisa.com) y unificar todos los archivos. Si hay redirección 301 entre ambos, asegurarse de que todo apunte al dominio canónico.

4. SITEMAPS
Lo que está bien
Sitemap Index correcto con 5 sub-sitemaps especializados.
Generación dinámica desde Supabase (vehículos, promociones, FAQs, páginas CMS).
Sub-sitemaps específicos: sitemap-pages.xml, sitemap-programmatic.xml, sitemap-faq.xml, sitemap-guides.xml, sitemap-vehicles.xml.
Ping automático a Google y Bing (IndexNow) cuando se publican nuevas FAQs.
Filtrado inteligente: solo incluye entidades published y slugs canónicos.
Problemas detectados
Problema	Severidad	Detalle
Sitemaps servidos como componente React	ALTO	Los sitemaps se renderizan como <pre> dentro de una SPA React. Los bots que no ejecutan JavaScript NO verán el contenido. El sitemap estático en public/sitemap.xml solo contiene el índice, no las URLs reales.
Falta <lastmod> en sitemap index	MEDIO	El sitemap index no incluye fechas de última modificación de cada sub-sitemap
robots.txt apunta a avisa.es/sitemap.xml	MEDIO	Incoherencia con el dominio del index.html (grupoavisa.com)
Recomendación crítica
Los sub-sitemaps (sitemap-pages.xml, sitemap-faq.xml, etc.) deben generarse como archivos estáticos o servirse desde un endpoint del servidor que devuelva Content-Type: application/xml. Actualmente dependen de que el bot renderice JavaScript, lo cual Google sí hace pero Bing y otros no siempre.

Problemas detectados
Problema	Severidad	Detalle
SPA sin SSR/SSG	ALTO	Todo el contenido se renderiza en el cliente. Aunque Google renderiza JS, el tiempo hasta el primer contenido visible (FCP) es más lento que con SSR. Los bots más simples no ven contenido.

Falta <noscript> en páginas internas	MEDIO	Solo el index.html tiene fallback noscript, las páginas internas de la SPA no
No hay service worker / PWA	BAJO	Sin cache offline ni instalación en dispositivo
assetsInlineLimit: 4096 puede ser agresivo	BAJO	Iconos SVG pequeños se inline como base64, aumentando el tamaño del bundle
